<objective>
Create the capability layer that v2.0 routes through: a typed registry for arbitrary `(op, provider)` pairs with constraints, cost/latency metadata, optional eval quality scores, and an `invoke` function contract.

This plan deliberately leaves existing `ImageProvider` and `ProviderRegistry` untouched. The capability registry is a second registry used by `image_op` and later `image_task`, while v1.0 tools continue using the existing provider interface.
</objective>

Implement `CapabilityRegistry` in `src/capabilities/registry.ts`.

Required API:
```ts
export class CapabilityRegistry {
  register(capability: Capability): void
  get(op: CapabilityOp, provider: string): Capability | undefined
  list(op?: CapabilityOp): Capability[]
  listByProvider(provider: string): Capability[]
  has(op: CapabilityOp, provider: string): boolean
}

export const capabilityRegistry = new CapabilityRegistry();
```

Implementation details:
- Store entries in `private capabilities = new Map<CapabilityKey, Capability>()`.
- Use key format `${op}:${provider}`.
- `register()` must support overwriting the same `(op, provider)` pair.
- If an existing capability exists for the same `(op, provider)` and its `modelVersion` differs from the incoming `modelVersion`, set incoming `quality` to `undefined` before storing.
- `list(op)` returns only capabilities for that op when `op` is provided; otherwise returns all capabilities.
- `listByProvider(provider)` filters by exact provider string.
- Do not import from `src/providers/index.ts`.

Update `src/capabilities/index.ts` to export from `types.ts`, `registry.ts`, and `register.ts`.
